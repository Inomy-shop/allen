# Curated And Mandatory Context

Allen uses two separate context collections for repo work:

- curated context for semantic recall, Cognee ingestion, graph building, and optional injection;
- mandatory context for exact always-load agent guidance.

They are intentionally separate. Curated context helps Allen find relevant task context. Mandatory context tells Allen what must always be injected for a specific agent.

## Curated Context

Curated context is stored in `repo_context_curation_entries` and is usually generated by `repo-context-curator`.

The curator reads git-tracked context files, skips unchanged files by hash, stages generated entries, validates coverage, then promotes valid entries to the final collection. Users can also add or edit entries from the Context Management UI.

Only active entries are ingested into Cognee:

```text
inclusion = include
```

### Curated Context Fields

| Field | Meaning | How to fill it |
|---|---|---|
| `entryId` | Stable identifier for the curated entry. | Let Allen generate it unless importing existing curated content. |
| `path` | Repo-relative source path or manual entry path. | Use exact repo paths when source-backed. Manual entries use a stable `manual/...` path. |
| `title` | Human-readable title shown in UI and diagnostics. | Use a specific name, not a broad label like "docs". |
| `sourceHash` | Hash of the source file or manual content marker. | Generated by curator; manual entries get a `manual:` hash. |
| `inclusion` | Whether the entry participates in context. | Use `include` for active entries, `exclude` for intentionally skipped files, `stale` for removed/obsolete source files. |
| `injectionPolicy` | How Allen may inject resolved content. | Use `snippet` for normal injectable curated context, `manifest_only` for searchable pointers, `never_full_auto` for entries that should not auto-inject. |
| `summary` | One-line preview for lists and diagnostics. | Summarize the useful content, not just the filename. |
| `curatedContext` | Agent-facing curated content. | Write faithful, source-grounded context that can be injected after retrieval. It should be useful to an agent without requiring the full original document. |
| `retrievalText` | Cognee-facing ingestion text. | Preserve entities, workflows, APIs, tables, decisions, constraints, aliases, and relationships for semantic recall and graph extraction. |
| `chunks` | Section-level retrieval/cognify units. | Use for large or multi-topic docs. Each chunk should stand alone with enough local context. |
| `diagnostics` | Validation or lifecycle notes. | Generated by Allen; use for debugging rather than authoring. |

### Curated Text Usage

`retrievalText` and `chunks[].text` are what Cognee should learn from during context builds. They should be rich enough for semantic search and graph extraction.

`curatedContext` is what Allen prefers for agent-facing injection after a Cognee result resolves back to the curated entry. It should be concise, faithful, and directly actionable.

`chunks` are not just tiny excerpts. For large documents, they are meaningful section-level context units so retrieval can land on the right part of the document without losing important constraints.

## Mandatory Context

Mandatory context is stored in `repo_mandatory_context_mappings` and is usually generated by `repo-mandatory-context-mapper`.

Mandatory mappings are loaded by exact Allen agent name. They bypass optional Cognee relevance thresholds because they are always-load instructions for that agent.

Use mandatory context only for narrow policies, rules, safety constraints, workflow contracts, and repo practices that the agent must know before it starts.

### Active And Inactive Mappings

Mappings use the `enabled` field as the source of truth for active/inactive state. At runtime Allen loads only enabled mappings; inactive rows are never injected.

When new mappings are saved for a repo and an agent (for example by the one-click setup flow), old active mappings for the same repo and agent are deactivated — but only after the new mappings have been saved successfully. If the save fails, existing active mappings remain unchanged.

Inactive rows are kept for audit and troubleshooting. They do not participate in runtime injection regardless of their other fields.

Optional audit fields are written when deactivation happens through the setup flow:

| Field | Meaning |
|---|---|
| `deactivatedAt` | Timestamp when the mapping was deactivated. |
| `deactivatedByRunId` | The setup run identifier that triggered deactivation. |
| `deactivationReason` | Reason for deactivation: `replaced` (a newer mapping superseded it) or `reviewed_empty` (the mapper reviewed the agent but produced no mappings). |

These fields are informational only and are not part of a versioning system.

### Mandatory Context Fields

| Field | Meaning | How to fill it |
|---|---|---|
| `mappingId` | Stable mapping identifier. | Let Allen generate it unless preserving an existing mapping. |
| `agentName` | Exact Allen agent that receives this context. | Must match an existing agent name, such as `backend-developer` or `code-reviewer`. |
| `title` | Human-readable title. | Name the rule or guideline clearly. |
| `content` | Full mandatory text injected for the agent. | Keep it exact and narrow. Do not paste broad docs unless every run for that agent needs them. |
| `sourcePath` | Repo-relative source path, if source-backed. | Use when the mandatory text comes from a repo file. |
| `sourceHash` | Hash of the source content, if available. | Generated by mapper or provided by an import. |
| `contentHash` | Hash of the injected content. | Generated by Allen. |
| `enabled` | Whether the mapping is active. | Disable instead of deleting when testing or temporarily suppressing context. |
| `reasoning` | Why this context is mandatory for the agent. | Explain the always-load need, not generic usefulness. |

## When To Use Which

Use curated context when:

- the content should be discoverable by task, role, or semantic search;
- Cognee should use it for recall, chunks, entities, or relationships;
- it is useful for some tasks but not always needed for every run of an agent.

Use mandatory context when:

- the same agent must always receive the content for this repo;
- missing it would likely cause unsafe, invalid, or non-compliant work;
- the content is short enough and important enough to inject before work starts.

Avoid making broad PRDs, full runbooks, large generated docs, or agent persona files mandatory. Curate those as searchable context unless the user explicitly wants them always injected.

## Context Management UI Workflow

In the repo **Context Management** page:

- use the **Setup** card above the tabs to run the full pipeline (curation, mandatory mapping, and graph refresh) in one coordinated action. The card shows phase-level progress and a resume action when a run fails.
- use **Curated Context** to review, edit, add, or mark curated entries;
- use **Mandatory Context** to view active mappings for all agents or filter by one agent. Enable **Show inactive (troubleshooting)** to include deactivated rows when diagnosing unexpected context injection behavior;
- use **Playground** to see mandatory refs, Cognee recalls, rerank scores, selected/rejected refs, resolved curated content, and injection decisions;
- use **Context Graph** to refresh or clean-build the Cognee context dataset from active curated entries.

After editing curated entries, run **Refresh Context** so Cognee and chunk-source mappings reflect the latest curated content. After editing mandatory mappings, new agent runs will use the updated mandatory content directly.
