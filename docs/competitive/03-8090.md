# 8090 AI (8090.ai) — Deep Dive

**Last reviewed:** 2026-04-17
**Category:** Enterprise AI-native SDLC orchestration (spec-rigorous)
**One-line pitch:** A three-module software factory (Refinery → Foundry → Planner) that binds PRDs, architectural blueprints, and implementation work orders together in a Knowledge Graph — aimed at regulated enterprises who need artifact-driven development.

---

## 1. Company & traction

- **Origin:** 8090 incubator founded by **Chamath Palihapitiya** (self-funded, no external LPs). Pivoted from incubator to operating company around the **Software Factory** product.
- **Founders / leadership:** Co-founder **Sina Sojoodi** (engineering leadership). Other founders not fully public.
- **Funding:** Chamath-funded; reported AI compute costs approaching ~$10M/year (per his LinkedIn / Substack).
- **Major customer / partnership:** **EY.ai PDLC** — Ernst & Young launched a co-branded product built on Software Factory (Mar 2026). Plan is to deploy to **tens of thousands of EY consultants** across US operations, supporting internal engineering AND client engagements.
- **Other customer detail:** not heavily publicized; 8090 is a more consulting-adjacent / enterprise-direct motion than a PLG play.

## 2. Product — Software Factory (three modules)

### 2a. Refinery (Requirements layer)
**Purpose:** Transform raw ideas, artifacts, and feedback into structured PRDs that serve as the single source of truth for product intent.

**Artifacts produced:**
- **Product Overview** — motivation, executive summary.
- **Feature Requirements** — feature-level requirements documents.
- Structured PRD with features, goals, success criteria.

**Behavior:** Refinery ingests fragments (Slack threads, tickets, docs, transcripts) and drives an interactive PRD-authoring loop. Agents surface gaps, inconsistencies, missing acceptance criteria.

### 2b. Foundry (Architecture layer)
**Purpose:** Transform PRDs into actionable technical blueprints that keep product intent, system design, and implemented code aligned over time.

**Artifacts produced:**
- **Blueprints** organized as a hierarchy of **Feature Nodes** — granular design decisions tied back to PRD requirements.
- **Foundations** — project-wide technical context: architectural principles, tech-stack decisions, security requirements, standards, deployment conventions.

**Key agent:**
- **Foundry Agent** — helps define the technical specifications for a feature.
- **Feature Extraction Agent** — powers Foundry's expansion of the PRD into a structured Blueprint document.

### 2c. Planner (Execution layer)
**Purpose:** Turn Blueprints into **Work Orders** — context-rich prompts that can be handed to coding agents or human engineers.

**Artifacts produced:**
- **Work Orders** — per-implementation-unit prompts packed with relevant PRD sections, blueprint fragments, code-search findings.
- Implementation plans that are codebase-aware.

**Key agent:**
- **Planner agent** with a **dedicated code-search sub-agent** that finds similar implementations and existing patterns in the user's codebase.
- **MCP integration** — Planner integrates with local development environments via MCP, so Work Orders can flow into Cursor / VS Code / Claude Code / etc.

## 3. The Knowledge Graph

The central architectural concept binding the three modules:

- **Unified context layer** — PRD sections, Feature Nodes, Work Orders, code references, and implementations are **nodes** in a graph with traceable edges.
- **Co-evolution** — when a PRD changes, downstream blueprints and work orders flag misalignment; when code changes, upstream blueprints update.
- This is the 8090 analog of Factory's HyperCode or Letta's memory blocks — but it's **artifact-level**, not runtime-state-level.

## 4. Agents & orchestration

- **Role-based** — PMs, designers, engineers, QA collaborate in the same workspace, each with AI teammates calibrated to their role.
- **Agents surface gaps, document reasoning, highlight tradeoffs, maintain coherence across artifacts.**
- **Multi-model orchestration** — explicitly markets that it orchestrates across multiple AI models; specific routing logic not published.
- **No published per-call model swap** — enterprise contract-driven; users don't pick models per call.
- Orchestration is **module-sequential** (PRD → Blueprint → Work Order) with **intra-module agent collaboration**.

## 5. Model plumbing

- **Multi-model** across multiple providers; specific list not public.
- **Routing** is internal — Work Orders are designed to be consumed by **external coding agents** (Cursor, Claude Code, etc.) via MCP, so 8090 is not aiming to be the coding agent itself.
- **AWS and Azure partnerships** implied by EY motion.

## 6. Human-in-the-loop

- **Artifact-driven** — every gate is a human-reviewable artifact (PRD, Blueprint, Work Order), not an ephemeral decision.
- **Module hand-offs** — PRD must be approved before Blueprint generation; Blueprint must be approved before Work Orders.
- **Comments, reviews, and edits** on each artifact type.
- Deep fit for regulated industries (finance, pharma, government) where audit trails matter.

## 7. Integrations

- **MCP-first for downstream coding** — Work Orders → MCP → Cursor / Claude Code / Windsurf / VS Code.
- **GitHub / GitLab** for codebase awareness.
- **Slack / Jira / Confluence** for ingestion into Refinery (presumed).
- **EY's internal systems** for the PDLC partnership.

## 8. Deployment / pricing

- **Cloud SaaS**, enterprise-only.
- **No published self-serve pricing.**
- **Enterprise contracts** with EY-style deployments (dedicated support, customer-specific integrations).
- **SOC-2 / compliance** implied by enterprise focus (not publicly confirmed).

## 9. Strengths

1. **Strongest upstream spec rigor in the cohort** — PRD → Blueprint → Work Order is the most opinionated and human-reviewable planning pipeline.
2. **Knowledge Graph keeps artifacts in sync** — when PRD changes, blueprints flag stale sections.
3. **Enterprise distribution via EY** — instant access to tens of thousands of consultants and their client engagements.
4. **Role-based collaboration** — PMs, designers, engineers, QA in one tool, not fragmented.
5. **Doesn't compete with coding agents** — hands off via MCP; complementary to Claude Code / Cursor / Factory.
6. **Chamath's backing** — deep pockets, long runway for enterprise sales cycles.
7. **Audit-friendly by design** — every decision is an artifact with traceable provenance.

## 10. Weaknesses

1. **Heavyweight process** — PRD/Blueprint/Work Order is overkill for startups or fast-iteration teams. Best fit is regulated enterprise.
2. **Closed SaaS** — no self-host, no OSS, no community extensibility beyond what the modules expose.
3. **Not a coding agent** — relies on external coding agents for implementation; if your org doesn't have one standardized, you still need to buy another tool.
4. **High burn rate** (~$10M/year AI compute per Chamath) suggests pricing has to be enterprise-premium to sustain.
5. **Small public customer list** — EY is the flagship; breadth of production deployments unclear.
6. **No published benchmarks** — unlike Factory (SWE-bench) or Kiro (AWS-published), 8090 leans on qualitative enterprise narrative.
7. **Documentation is thin** for external developers — the docs site covers modules but not extensibility / API surface.

## 11. Who it's for

- **Fortune 500 modernization programs** — legacy SAP / mainframe replacements, regulated finance/pharma/government SDLC.
- **Large consulting practices** — EY is the archetype; Deloitte / Accenture-style firms are natural fits.
- **Enterprises with heavy PM/architect/audit functions** that already produce PRDs and blueprints — 8090 makes that process 10× faster rather than inventing a new process.
