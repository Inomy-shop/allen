## Core Concepts

### V2 Schema Architecture

The schema system has three layers:

| Layer | Contains | Location | Has `enum_values`? | Who Manages |
|-------|----------|----------|--------------------|-------------|
| **Base Schema** | Common fields for ALL products (name, brand, model, price, color, upc, etc.) | MongoDB `base_schemas` | No | Shared across categories |
| **Base Specifications** | Category-specific overrides for base schema fields (importance, extraction_rules, validation_rules ONLY) | MongoDB `product_schemas` ‚Üí `base_specifications` | **NO ‚Äî never** | This agent (Workflow C) |
| **Specifications** | Category-specific technical specs with full field definitions including enum_values | MongoDB `product_schemas` ‚Üí `specifications` | **YES ‚Äî required for keyword/keyword[] fields** | This agent (Workflow A/B) |

**CRITICAL RULES**:
- Never include base schema fields in specifications. Always verify the current list from `GET /api/schemas/base/active`.
- `base_specifications` and `specifications` have DIFFERENT field shapes ‚Äî see below.

### Field Definitions: Specifications vs Base Specifications

**`specifications` fields (Workflow A/B)** ‚Äî full V2 definition WITH enum_values:

```typescript
// For category-specific specification fields
interface SpecificationFieldV2 {
  field_type: FieldType | 'keyword[]' | 'integer[]';
  required: boolean;
  extraction_rules: string;
  validation_rules: string;
  importance: 'CRITICAL' | 'RECOMMENDED' | 'OPTIONAL';
  enum_values?: string[];        // REQUIRED for keyword/keyword[] fields
}
```

**`base_specifications` fields (Workflow C)** ‚Äî same shape but WITHOUT enum_values:

```typescript
// For base schema field overrides ‚Äî NO enum_values allowed
interface BaseSpecificationField {
  field_type: FieldType;          // COPIED from base schema ‚Äî DO NOT change
  required: boolean;              // COPIED from base schema ‚Äî DO NOT change
  extraction_rules: string;       // Category-specific ‚Äî generated by agent
  validation_rules: string;       // Category-specific ‚Äî generated by agent
  importance: 'CRITICAL' | 'RECOMMENDED' | 'OPTIONAL';  // Category-specific
  // NO enum_values ‚Äî base fields are shared, enum constraints go in specifications
}
```

**Why the difference**: Base schema fields (brand, color, price, etc.) are shared across ALL categories. Their valid values are universal. Category-specific enum constraints belong only in `specifications`.

**`enum_values` rules:**
- **REQUIRED** for `keyword` and `keyword[]` fields that have a finite set of valid values
- **NOT used** for `integer`, `float`, `boolean` fields
- Must be **exhaustive** ‚Äî merge your domain knowledge with actual values from `product` table queries
- Include normalized forms only (e.g., `"IPS"` not `"In-Plane Switching"`) ‚Äî put normalization mappings in `extraction_rules`

### Field Type Decision Guide

| Type | Use When | enum_values? | Examples (across verticals) |
|------|----------|-------------|---------|
| `integer` | Whole numbers ‚Äî counts, capacities, specs in whole units | No | `ram_gb`, `weight_capacity_lbs`, `thread_count`, `burner_count`, `shelf_count` |
| `float` | Decimal measurements ONLY | No | `screen_size_inches`, `width_inches`, `fabric_weight_gsm`, `blade_diameter_inches` |
| `boolean` | Yes/no feature flags | No | `has_touchscreen`, `assembly_required`, `is_machine_washable`, `is_outdoor_rated` |
| `keyword` | Any string field ‚Äî single value. If finite set, provide `enum_values` | **Yes** (if finite set) | `panel_type`, `frame_material`, `processor`, `fabric_composition`, `fuel_type`, `pattern_description` |
| `keyword[]` | Any string array ‚Äî product can have MULTIPLE values | **Yes** (if finite set) | `connectivity_options`, `resistance_types`, `care_instructions`, `compatible_surfaces` |
| `integer[]` | Array of whole numbers (rare) | No | `supported_ram_speeds_mhz`, `available_lengths_inches` |

**CRITICAL: NEVER use `text` as a field_type.** All string fields MUST be `keyword` (single) or `keyword[]` (array). The `text` type is NOT allowed in product schemas.

**Decision tree for string fields:**
- Can you list ALL possible values (< 30 distinct)? ‚Üí `keyword` or `keyword[]` with `enum_values`
- Sort of (30-100 known values)? ‚Üí `keyword` with `enum_values` listing known values + note in validation_rules "allow other recognized values"
- No (unbounded/too varied)? ‚Üí `keyword` WITHOUT `enum_values` ‚Äî use pattern-based validation in `validation_rules` instead

### Importance Levels (Customer-Centric Assignment)

For EACH field, ask in order:
1. "Would a customer RETURN the product if this spec is wrong?" ‚Üí **CRITICAL**
2. "Would a customer COMPARE this spec when shopping?" ‚Üí **RECOMMENDED**
3. "Would a customer even NOTICE if slightly off or missing?" ‚Üí NO = **OPTIONAL**

**CRITICAL: Importance depends on the product vertical, NOT a universal electronics default.**

| Vertical | CRITICAL examples | RECOMMENDED examples | OPTIONAL examples |
|----------|-------------------|---------------------|-------------------|
| **Electronics** (laptops, phones) | processor, ram_gb, storage_gb, screen_size | refresh_rate, weight, battery_life | max_ram, certifications, led_indicators |
| **Appliances** (washers, fridges) | capacity_cu_ft, energy_rating, cycle_types | noise_level_db, dimensions, smart_features | certifications, cord_length |
| **Furniture** (sofas, desks, beds) | **width_inches, depth_inches, height_inches**, frame_material, weight_capacity_lbs | upholstery_material, finish_type, assembly_required | packaging_box_count, tools_required |
| **Fashion** (clothing, shoes) | **size, fit_type**, fabric_composition | care_instructions, closure_type, pattern | season, layering_potential |
| **Garden/Outdoor** (mowers, grills) | cutting_width_inches, fuel_type, engine_power | blade_type, wheel_size, assembly_required | warranty_type, shipping_weight |
| **Home Improvement** (tools, plumbing) | power_watts, voltage, chuck_size | speed_settings, cord_length, weight | certifications, included_accessories |
| **Baby/Kids** (strollers, car seats) | **weight_limit_lbs, age_range**, safety_certifications | fold_type, wheel_type, canopy_type | cup_holder_count, storage_pockets |

**Key rule:** Dimensions (W√óD√óH) are OPTIONAL for electronics but **CRITICAL for furniture, fashion, and any product where physical fit matters.** Always think from the customer's perspective for the SPECIFIC product type.

### Step A2: Generate Schema

Generate the complete `specifications` JSON. Follow these rules:

**Structure:**
- 3-5 logical groups (e.g., `display`, `performance`, `connectivity`, `physical`)
- 15-25 specification fields total
- Each field has these V2 properties: `field_type`, `required`, `extraction_rules`, `validation_rules`, `importance`, and `enum_values` (for keyword/keyword[] fields)

**Field Naming ‚Äî REUSE EXISTING FIELDS (CRITICAL FOR OPENSEARCH):**

Every field name in `specifications` creates a mapping in the OpenSearch `unified_product_index_v2`. Duplicate names for the same concept (e.g., `panel_type` in one category and `tv_panel` in another) bloat the index with redundant mappings. This is a **hard requirement**, not a suggestion.

**Process (MUST follow for every field):**

1. Load ALL existing field paths from Step A1: `GET /api/schemas/products` ‚Üí extract all `specifications.*` field names with their types
2. For EACH field you want to create, scan the existing list:
   - If an existing field represents the **same concept** ‚Üí REUSE IT EXACTLY (same name, same type)
   - If uncertain between "reuse" vs "new" ‚Üí CHOOSE REUSE
   - Only create a NEW field name if NO existing field matches the concept
3. When reusing a field, you MUST keep the SAME `field_type` as the existing field

**Common duplicate traps to avoid:**

| You want to create | But this already exists | USE THIS |
|---------------------|----------------------|----------|
| `tv_panel` | `panel_type` | `panel_type` |
| `screen_diagonal` | `screen_size_inches` | `screen_size_inches` |
| `product_weight` | `weight_lbs` | `weight_lbs` |
| `sofa_width` | `width_inches` | `width_inches` |
| `wifi_version` | `wifi_standard` | whichever exists |
| `material_type` | `frame_material` | check if they're the same concept |
| `washing_capacity` | `capacity_cu_ft` | `capacity_cu_ft` (if same unit) |

**Naming conventions:**
- Use `snake_case` for all field names
- Include units when applicable: `weight_lbs`, `screen_size_inches`, `width_inches`, `capacity_cu_ft`, `fabric_weight_gsm`
- Use GENERIC names that work across categories, not category-specific prefixes (e.g., `panel_type` not `monitor_panel_type`)

**Data Source Priority (from the `product` table):**

The `product` table has these key columns that the extraction LLM reads:
- `name` ‚Äî product title (MOST spec data lives here, e.g., "Dell 27\" IPS 165Hz QHD Monitor")
- `brief_description` ‚Äî short description
- `details.features` ‚Äî bullet point features array
- `details.specifications` ‚Äî structured vendor specs (key-value JSONB)
- `details.*` ‚Äî other vendor-specific fields (brand, dimensions, weight, pricing, etc.)

**Extraction priority (highest to lowest):** `name > brief_description > features > details.specifications > details.*`

**Extraction Rules ‚Äî Be VERY Specific:**

Write extraction rules as clear instructions the extraction LLM will follow. Include:
1. **Location hints**: "Look in name (title) first, then details.specifications under 'X', then brief_description"
2. **Data source priority**: Always state `name > brief_description > features > details.specifications > details.*`
3. **Patterns**: Actual patterns from real product data for this specific product type
4. **Synonyms**: Alternative terms vendors use for the same concept
5. **Unit handling**: Conversion instructions when applicable
6. **Normalization** (for keyword fields): Mappings from vendor terms to normalized enum values
7. **Multiple values**: Which value to pick when multiple exist

Examples across verticals:
- **Electronics**: "Look in title for screen size patterns: '15.6\"', '15.6-inch'. Check details.specifications under 'Screen Size'."
- **Furniture**: "Look in title for dimensions. Check details.specifications under 'Overall Width', 'Product Width'. Convert cm to inches if needed."
- **Fashion**: "Look in title and details.specifications under 'Fabric Type', 'Material', 'Fabric Content'. Normalize: '100% cotton' ‚Üí 'Cotton'."
- **Appliance**: "Look in title for capacity patterns: '4.5 cu. ft.', '4.5 cubic feet'. Check details.specifications under 'Washer Capacity'."

**Validation Rules:**

For numeric fields, include concrete ranges specific to the product type:
- Electronics: "Must be between 10-18 inches for laptops. Common values: 13.3, 14, 15.6, 16, 17.3"
- Furniture: "Width must be between 20-120 inches for sofas. Common values: 72, 84, 96 (6ft, 7ft, 8ft)"
- Appliance: "Capacity must be between 1.0-6.0 cu ft for washers. Common values: 4.2, 4.5, 5.0, 5.2"
- Include unit conversion guidance when applicable

For keyword/keyword[] fields, the exhaustive values go in `enum_values` (not validation_rules). Use validation_rules for:
- Normalization guidance: "Case-insensitive match, normalize to uppercase abbreviation form"
- Edge case handling: "If value contains a valid type as substring (e.g., 'Fast IPS'), extract the type ('IPS')"
- Reject rules: "Reject values not found in enum_values list"

**enum_values ‚Äî Build Exhaustive Lists:**

For each keyword/keyword[] field:
1. Start with your domain knowledge of industry-standard values
2. Query `product` table to discover actual values vendors use
3. Merge both into a deduplicated, normalized list
4. Include ONLY normalized forms (e.g., `["IPS", "VA", "TN", "OLED"]` ‚Äî put mappings like "In-Plane Switching ‚Üí IPS" in extraction_rules)

**MUST NOT include base schema fields** ‚Äî verify from `GET /api/schemas/base/active`.
